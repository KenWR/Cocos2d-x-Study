# Process

이 문서는 `Cocos2d-x-Study` 저장소의 진행 상황과 다음 작업을 기록하는 작업 로그다.

## 현재 목표

`PoopDodge`를 Cocos2d-x 4.0 기반 C++ 피하기 게임으로 만들고, Windows 실행 확인부터 Android 빌드와 WebGL 포팅 가능성까지 순서대로 실험한다.

## 진행 완료

### 저장소 구성

- Cocos2d-x 4.0 엔진 소스를 저장소 안에 포함했다.
- 예제 프로젝트 `PoopDodge`를 `cocos2d-x/projects/PoopDodge` 경로에 추가했다.
- 빌드 산출물이 Git에 섞이지 않도록 `.gitignore`를 정리했다.

### Windows Debug 빌드 안정화

- Visual Studio 2022와 CMake 환경에서 Windows Debug 빌드를 확인했다.
- Cocos2d-x 4.0의 Windows prebuilt 라이브러리에 맞춰 Win32(x86) 빌드 경로를 기준으로 잡았다.
- Win64/x64 빌드는 사용하지 않기로 하고, 예전 `build` 폴더를 제거했다.
- 최신 Windows SDK/MSVC 환경에서 발생한 `GWL_*` 상수 문제를 `GWLP_*` 상수로 수정했다.
- MSVC PDB 동시 쓰기 오류를 피하기 위해 `/MP` 대신 `/FS` 옵션을 사용하도록 조정했다.
- 빌드 오류 원인과 해결 과정을 `docs/build_error.md`에 정리했다.

### 첫 플레이 가능한 게임 구현

- 기본 `HelloWorld` 예제 이미지 화면을 제거했다.
- 기본 창 크기를 `960 x 640`으로 키웠다.
- 단색 배경, 타원형 플레이어, 갈색 삼각형 장애물을 Cocos2d-x 도형으로 표시했다.
- `Enter` 키로 게임을 시작하고 다시 시작할 수 있게 했다.
- 화면의 `Start` / `Retry` 버튼으로도 시작과 다시 시작을 할 수 있게 했다.
- 좌우 방향키 또는 A/D 키로 플레이어를 움직이게 했다.
- 장애물이 위에서 아래로 떨어지고, 화면 밖으로 나가면 다시 위에서 떨어지게 했다.
- 생존 시간이 길어질수록 장애물 낙하 속도와 활성 개수가 늘어나게 했다.
- 플레이어와 장애물이 충돌하면 게임 오버 상태로 바뀌게 했다.
- 생존 시간을 점수로 표시했다.

### 문서 정리

- README에 개발 환경, 빌드 방법, 실행 방법, 플레이 방법을 정리했다.
- `docs/Process.md`에 작업 진행도와 다음 작업을 기록하기 시작했다.

### 빌드 실행 단순화

- 저장소 루트에서 `build-win32.cmd`를 실행하면 Win32 Debug 빌드가 되도록 스크립트를 추가했다.
- PowerShell 전용 스크립트 대신 Windows `cmd`에서 바로 실행할 수 있는 `.cmd` 파일을 사용한다.

### 실행 확인

- Windows Win32 빌드 결과물을 직접 실행해 현재 게임 흐름을 확인했다.

## 현재 상태

- Windows Debug 빌드는 `-A Win32` 옵션을 사용했을 때 통과한다.
- 현재 유지하는 Windows 빌드 폴더는 `build-win32`다.
- Win64/x64 빌드 폴더는 사용하지 않는다.
- `PoopDodge`는 외부 이미지 없이 도형만 사용한 첫 플레이 가능한 상태다.
- 아직 파일명과 클래스명은 Cocos2d-x 기본 예제의 `HelloWorldScene` / `HelloWorld`를 유지하고 있다.
- 실제 플레이 감각, 난이도 증가 폭, 충돌 범위는 직접 실행하면서 조정해야 한다.

## 다음 작업

### 1. 조작감과 난이도 조정

- 플레이어 이동 속도와 크기를 조정한다.
- 장애물 크기, 시작 속도, 속도 증가량을 조정한다.
- 장애물 개수 증가 간격과 최대 개수를 조정한다.
- 충돌 범위가 너무 넓거나 좁지 않은지 확인한다.

### 2. 코드 이름 정리

- `HelloWorldScene` 파일명과 `HelloWorld` 클래스명을 게임 이름에 맞게 바꿀지 결정한다.
- 이름을 바꾼다면 CMake 소스 목록과 include 경로까지 함께 정리한다.

### 3. Windows 실행 경험 정리

- README의 빌드/실행 명령이 실제 작업 흐름과 맞는지 계속 갱신한다.
- 필요하면 실행까지 함께 처리하는 별도 스크립트를 추가한다.
- 필요하면 실행 화면 캡처나 조작 설명을 별도 문서로 분리한다.

### 4. Android 빌드 포팅

- Cocos2d-x 4.0의 Android 빌드 요구사항을 확인한다.
- 현재 Windows 개발 환경에서 필요한 Android SDK, NDK, Gradle 버전을 정리한다.
- Windows 빌드와 충돌하지 않도록 Android 빌드 절차를 별도 문서로 분리한다.
- Win32에서 확인한 게임 흐름이 Android에서도 동일하게 동작하는지 확인한다.

### 5. WebGL 포팅 가능성 확인

- Cocos2d-x 4.0에서 WebGL 또는 웹 타깃을 직접 지원하는지 확인한다.
- 직접 지원이 어렵다면 Emscripten 기반 포팅 가능성을 조사한다.
- 포팅 비용이 크면 `PoopDodge` 게임 로직을 별도 웹 구현으로 옮기는 대안을 검토한다.
- Win32 게임 흐름을 기준으로 웹에서 재현할 수 있는 범위를 정한다.

## 작업 원칙

- 먼저 Windows에서 작게 동작하는 게임을 완성한다.
- 문제를 만나면 원인, 증상, 해결 방법을 `docs`에 남긴다.
- README는 실행 방법 중심으로 짧게 유지하고, 자세한 과정은 `docs`에 분리한다.
